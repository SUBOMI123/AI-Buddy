<objective>
Add OS-native active application detection to the overlay. When the overlay opens, a non-blocking Rust command reads the frontmost app name (e.g., "Code", "Terminal", "Figma") via `active-win-pos-rs` and stores it in a SolidJS signal. The detected name is injected into the Claude system prompt and recorded in the memory DB as `appContext`.

Purpose: The AI can give context-aware guidance ("In VS Code, press Cmd+P to...") without relying on screenshot analysis for app identification. This satisfies CTX-01 (detection), CTX-02 (context enrichment), and CTX-03 (OS-sourced, no screenshot).

Output: New `app_context.rs`, updated `lib.rs`, `tauri.ts`, `ai.ts`, and `SidebarShell.tsx`. Compiles with `cargo build` and `npm run build`.
</objective>

```rust
// src-tauri/src/app_context.rs
// Phase 8: CTX-01, CTX-03 — OS-native active application detection
// Source: active-win-pos-rs 0.10.0 docs + 08-RESEARCH.md Pattern 1

use active_win_pos_rs::get_active_window;

/// Returns the bundle/process name of the frontmost application at time of call.
/// Uses OS-native API: NSWorkspace on macOS, GetForegroundWindow on Windows.
///
/// Returns Ok(Some(name)) on success.
/// Returns Ok(None) if no window has focus, app_name is empty, or detection fails.
///
/// IMPORTANT: Uses only `app_name` — does NOT read `title`.
/// On macOS, `title` requires Screen Recording permission and returns "" without it.
/// `app_name` is available without any special permission on macOS Sequoia (Pitfall 5).
///
/// Intentionally synchronous — get_active_window() is a fast synchronous OS call.
/// Do NOT add async. The caller (JS) calls this non-blocking (fire-and-forget).
///
/// CTX-03: No screenshot analysis involved. OS process table only.
#[tauri::command]
pub fn cmd_get_active_app() -> Result<Option<String>, String> {
    match get_active_window() {
        Ok(win) => {
            // Trim whitespace; cap at 100 chars to prevent prompt bloat from adversarial names
            let name = win.app_name.trim().chars().take(100).collect::<String>();
            if name.is_empty() {
                Ok(None)
            } else {
                Ok(Some(name))
            }
        }
        // Best-effort: never fail hard on app detection (Pitfall 3: Windows elevated process)
        Err(_) => Ok(None),
    }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS process table → app_name string | app_name is user-controlled (any process can set any name); crosses from OS into prompt injection |
| Rust → JS (IPC) | cmd_get_active_app return value crosses Tauri IPC boundary |
| JS → Claude API (via Worker) | appContext string is injected into system prompt sent to Claude |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-02-01 | Tampering | app_context.rs → system prompt | mitigate | `app_name` is trimmed and capped at 100 chars in `cmd_get_active_app` before returning. Claude does not execute prompt strings; injection risk is cosmetic only. Cap prevents prompt bloat from adversarially-named processes. |
| T-08-02-02 | Information Disclosure | active-win-pos-rs `title` field | mitigate | Phase 8 explicitly reads only `win.app_name`, never `win.title`. The `title` field can contain sensitive document names (e.g., "salary_2025.xlsx — Excel"). Enforced by code: no `.title` access in app_context.rs. |
| T-08-02-03 | Denial of Service | getActiveApp() blocking overlay | mitigate | Called fire-and-forget (`.then().catch()`), never awaited in the main overlay-shown handler. OS call failure returns None, not an error to the user. Overlay remains interactive regardless of detection result. |
| T-08-02-04 | Elevation of Privilege | Windows elevated process app_name | accept | On Windows, UAC-elevated processes return empty app_name. This is handled by returning Ok(None) — no data leak, no crash. Acceptable: app detection is best-effort enrichment, not a security gate. |
</threat_model>
