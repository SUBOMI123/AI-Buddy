<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User text input → sessionHistory | User's raw intent stored as session state and re-sent to Claude |
| sessionHistory → Claude API | Prior turn text included in outgoing messages array |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-03-01 | Tampering | lastIntent rendered in TaskHeaderStrip | accept | Rendered as SolidJS JSX text node — no innerHTML. User controls this value (it is their own typed intent). No injection surface. |
| T-09-03-02 | Denial of Service | sessionHistory unbounded growth | mitigate | `setSessionHistory` functional updater caps at 3 entries via `.slice(updated.length - 3)` in `onDone`. Enforced on every append. No bypass path in the normal UI flow. |
| T-09-03-03 | Information Disclosure | sessionHistory in-memory | accept | D-10: never serialized to localStorage, disk, or network except as part of the Worker `/chat` request (same trust boundary as existing single-turn requests). Clears on app restart and explicit "New task" action. No cross-session or cross-user leakage risk. |
| T-09-03-04 | Spoofing | "New task" reset without confirmation | accept | D-01 explicitly locks this as no-confirmation. In-memory only data loss is low-stakes — no files or persistent data affected. User can re-submit their intent. ASVS L1 has no requirement for confirmation on in-memory state resets. |
</threat_model>
