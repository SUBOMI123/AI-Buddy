## Interaction Contracts

### Submit Flow (CORE-01, CORE-02, CORE-04)

| Step | UI Behavior |
|------|-------------|
| User types intent and presses Enter (or clicks send icon) | 1. Previous guidance clears immediately (D-07). 2. LoadingDots appears in the content area. 3. TextInput clears and remains enabled for next question. |
| Screenshot capture begins | Happens silently -- no visible indicator. Capture takes ~50-200ms (D-01). |
| Screenshot capture fails | No modal or blocking UI. A small inline notice appears above the guidance area: "Screen capture unavailable -- guidance may be less specific" (D-04). Request continues as text-only. |
| First token arrives from Claude | LoadingDots disappears. Streaming text begins appearing in the content area (D-05, D-06). |
| Tokens stream in progressively | Text appends word-by-word in the guidance area. Content area auto-scrolls to keep the latest text visible. |
| Stream completes | No additional visual change. Final text remains displayed until next submit. |
| User submits new question while stream is active | In-flight request is aborted (AbortController). Flow restarts from step 1. |

### Loading State (D-06)

| Property | Value |
|----------|-------|
| Indicator | Three pulsing dots: `...` |
| Dot color | `var(--color-text-secondary)` |
| Animation | Opacity pulse 0.3 to 1.0, staggered 200ms per dot, 1.2s cycle |
| Position | Centered vertically in the content area (same position as EmptyState) |
| Duration | Visible from submit until first token arrives (~2-3 seconds typical) |
| Disappearance | Instant removal -- no fade-out -- replaced by first streaming text |

### Streaming Display (D-05, D-07)

| Property | Value |
|----------|-------|
| Text container | Full width of content area (sidebar width minus 2x md padding = 268px) |
| Font | Body: 14px, weight 400, line-height 1.5 |
| Color | `var(--color-text-primary)` |
| Rendering | Raw text during stream -- no step splitting while streaming. Claude's numbered output renders as plain text with line breaks preserved. |
| Scroll behavior | Content area scrolls to bottom on each new token if user has not manually scrolled up. If user scrolls up, auto-scroll pauses. Resumes when user scrolls back to bottom. |
| Clear behavior | On new submit, entire content area clears and LoadingDots replaces it (D-07) |

### Error State (D-11)

| Property | Value |
|----------|-------|
| Position | Inline in the guidance content area (same position where streaming text appears) |
| Layout | Centered vertically, left-aligned text |
| Error message | Body size (14px), `var(--color-text-primary)` |
| Retry button | Text-only button, accent color, Body size, 44px minimum touch target |
| Retry behavior | Clicking Retry re-submits the same intent (re-captures screenshot) |
| No modal | Errors never use modal dialogs -- always inline (D-11) |

### Screenshot Fallback Notice (D-04, D-12)

| Property | Value |
|----------|-------|
| Position | Top of content area, above guidance text or loading dots |
| Style | Label size (12px), `var(--color-text-secondary)`, left-aligned |
| Copy | "Screen capture unavailable -- guidance may be less specific" |
| Persistence | Stays visible for the duration of the current guidance response. Clears on next submit. |
| No blocking | User flow continues -- this is informational only |

## Layout Specification

No layout changes from Phase 1. The sidebar remains 300px wide, docked to screen edge. The content area between DragHandle and TextInput now cycles through these mutually exclusive states:

```
+--[SidebarShell (300px)]--+
| [DragHandle]              |
|                           |
| State A: [EmptyState]     |  -- no guidance yet
| State B: [LoadingDots]    |  -- waiting for first token
| State C: [StreamingText]  |  -- tokens flowing in
| State D: [ErrorState]     |  -- request failed
|                           |
| [TextInput]               |
+---------------------------+
```

Optional overlay on any state:
- `[ScreenshotFallbackNotice]` -- appears at top of content area when screenshot capture fails, above whatever state is showing (B, C, or D)

## State Machine

SidebarShell content area follows this state machine:

```
[permission_needed] --> (grant) --> [empty]
[permission_needed] --> (dismiss) --> [no_permission]

[empty] --> (submit) --> [loading]
[no_permission] --> (submit) --> [loading]  (text-only, with fallback notice)

[loading] --> (first_token) --> [streaming]
[loading] --> (error) --> [error]

[streaming] --> (done) --> [streaming]  (final state, text stays)
[streaming] --> (submit) --> [loading]  (abort previous, restart)
[streaming] --> (error) --> [error]

[error] --> (retry) --> [loading]  (re-submit same intent)
[error] --> (submit) --> [loading]  (new intent)
```

TextInput remains enabled in all states except `permission_needed`.
