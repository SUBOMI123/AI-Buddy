Certificate import step (Pattern 6 from research):
```yaml
- name: Import Apple Developer Certificate
  env:
    APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
    APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}
    KEYCHAIN_PASSWORD: ${{ secrets.KEYCHAIN_PASSWORD }}
  run: |
    CERT_PATH="$RUNNER_TEMP/build_certificate.p12"
    KEYCHAIN_PATH="$RUNNER_TEMP/app-signing.keychain-db"
    echo "$APPLE_CERTIFICATE" | base64 --decode > "$CERT_PATH"
    security create-keychain -p "$KEYCHAIN_PASSWORD" "$KEYCHAIN_PATH"
    security set-keychain-settings -lut 21600 "$KEYCHAIN_PATH"
    security unlock-keychain -p "$KEYCHAIN_PASSWORD" "$KEYCHAIN_PATH"
    security import "$CERT_PATH" -P "$APPLE_CERTIFICATE_PASSWORD" -A -t cert -f pkcs12 -k "$KEYCHAIN_PATH"
    security set-key-partition-list -S apple-tool:,apple:,codesign: -s -k "$KEYCHAIN_PASSWORD" "$KEYCHAIN_PATH"
    security list-keychain -d user -s "$KEYCHAIN_PATH"
```

        run: npm ci

        env:
          APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
          APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}

**Important notes on the workflow:**
- `releaseDraft: true` creates a draft release — you must manually publish it after verifying all three job artifacts have uploaded. This is intentional: do NOT auto-publish until you've verified the artifacts.
- `TAURI_BUNDLER_DMG_IGNORE_CI: "false"` disables the Finder-based DMG window customization in CI (which times out), while still producing a valid DMG. This is the correct value per tauri-bundler 2.2.0+.
- `WORKER_URL` is hardcoded (not a secret) — it is a public URL, same as in `.cargo/config.toml`.
- The Windows job has no `args:` (no `--target` flag) — it builds for the native runner architecture (`x86_64-pc-windows-msvc`) by default.
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` is included even if blank — tauri-action handles an empty string gracefully.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions → macOS runner | Secrets injected as env vars; must never be echoed to logs |
| CI runner → Apple notarization API | APPLE_PASSWORD (app-specific password) transmitted to Apple servers |
| CI runner → GitHub Releases | GITHUB_TOKEN used to publish artifacts; scoped to this repo |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-02-01 | Information Disclosure | TAURI_SIGNING_PRIVATE_KEY in CI logs | mitigate | Secret injected via `secrets.TAURI_SIGNING_PRIVATE_KEY`; GitHub Actions automatically masks secret values in all log output; no `echo` of secret values in any run step |
| T-15-02-02 | Information Disclosure | APPLE_CERTIFICATE / APPLE_PASSWORD in CI logs | mitigate | Both injected via secrets; certificate decoded to temp file (`$RUNNER_TEMP/`) not workspace — file is not committed; keychain deleted after build by runner cleanup |
| T-15-02-03 | Tampering | Supply chain attack via tauri-action | mitigate | Pinned to `tauri-apps/tauri-action@v0` (official Tauri org action); no third-party forks; review action source if security posture requires hash pinning (future: use commit SHA pin) |
| T-15-02-04 | Information Disclosure | APP_HMAC_SECRET compiled into binary | accept | Compile-time embedding accepted for beta; low-value target; mitigate in v1.0 with server-side validation |
| T-15-02-05 | Tampering | Unauthorized tag push triggers release | mitigate | Only maintainers with write access to the repo can push tags; `releaseDraft: true` requires manual publish step — no auto-publish to public users |
| T-15-02-06 | Elevation of Privilege | Malicious GITHUB_TOKEN scope escalation | accept | GITHUB_TOKEN is auto-scoped to the repo; least-privilege by default in GitHub Actions; no additional permissions granted |
</threat_model>
