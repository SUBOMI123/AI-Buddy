# Feature Landscape: v3.0 Closed Beta Distribution

**Domain:** Desktop app beta distribution (Tauri v2, macOS + Windows)
**Researched:** 2026-04-13
**Milestone scope:** Closed beta only — not public launch, not App Store submission

---

## Scope Note

This file covers the v3.0 "Ship" milestone only. All v2.0 UX feature research (action-first UI, step tracking, multi-monitor, etc.) was captured in the previous iteration of this file. v3.0 is entirely about getting a working build into the hands of closed beta users on macOS and Windows.

---

## Table Stakes

Features that must exist before handing a download link to a single beta user. Missing any one = broken beta.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Cloudflare Worker deployed to production | App is non-functional without it; every AI, STT, and TTS call routes through the proxy. Users see a blank overlay if the worker is not live. | Low | KV namespace ID in wrangler.toml and all API key secrets are already documented as "PRODUCTION REQUIRED" in the codebase. Deploy is a one-command `wrangler deploy` once env vars and secrets are configured via `wrangler secret put`. The hard work (Hono routing, HMAC auth) is already built. |
| macOS code signing + notarization | macOS Gatekeeper blocks unsigned apps entirely on macOS 10.15+. Users double-click the app and see "Cannot be opened because it is from an unidentified developer." This is not a warning they can bypass without CLI tricks — it is a hard stop for non-technical users. Notarization is mandatory for Developer ID distribution outside the App Store. | Medium | Requires Apple Developer Program membership (~$99/yr). Certificate type: "Developer ID Application" (not "Apple Distribution," which is for App Store only). Tauri v2 handles the notarization upload automatically during build. Requires: APPLE_CERTIFICATE (base64 .p12), APPLE_CERTIFICATE_PASSWORD, and either APPLE_API_ISSUER + APPLE_API_KEY (preferred for CI — no expiring passwords) or APPLE_ID + APPLE_PASSWORD + APPLE_TEAM_ID. Notarization adds 2–5 min to macOS CI build time. No workaround exists for this wait — Apple scans the binary server-side. |
| Windows code signing | Windows SmartScreen shows a red "Unknown publisher — Windows protected your PC" warning on unsigned installers. Most non-technical users click "Don't run" by default. Effectively an install failure. | Medium | Since June 2023, all code signing private keys must be stored on hardware security modules — certificate authorities no longer issue exportable OV/EV keys. Options: (1) Microsoft Trusted Signing — $9.99/month, no registered business required, requires Azure account, integrates with tauri-action via trusted-signing-cli. This is the right choice for a closed beta with a small user count. (2) Azure Key Vault + EV cert — ~$300-500/yr, requires registered business. (3) OV cert on hardware token — similar cost, builds SmartScreen reputation slowly. Note: as of March 2024, EV certificates no longer grant instant SmartScreen reputation bypass — all new apps build reputation over time through download volume. For a closed beta with tens of users, SmartScreen warnings will still appear even after signing. Brief disclosure in your beta invite ("If Windows shows a warning, click 'More info' → 'Run anyway'") is the pragmatic mitigation. |
| Auto-updater configured and functional | Beta users will not manually reinstall to receive fixes. Every bug fix that requires emailing a new download link loses users. The auto-updater is not a nice-to-have — it is the mechanism that makes beta iteration possible. | Medium | Tauri v2 uses tauri-plugin-updater. Three things must be in place before the first build: (1) `createUpdaterArtifacts: true` in bundle config so Tauri generates the .sig signature files alongside installers. (2) Updater keypair: run `cargo tauri signer generate` to create public + private key. Place pubkey in tauri.conf.json `plugins.updater.pubkey`. Place private key in TAURI_SIGNING_PRIVATE_KEY CI secret. The private key is a one-time artifact — loss means inability to push updates to installed copies. Store it in a password manager immediately. (3) Live update endpoint: the simplest approach is GitHub Releases with `latest.json` generated by tauri-action (`includeUpdaterJson: true`). Point `endpoints` in tauri.conf.json at `https://github.com/<org>/<repo>/releases/latest/download/latest.json`. The `{{target}}` and `{{arch}}` variables are interpolated by the app at runtime. |
| Signed reproducible CI build pipeline | Manual local builds cannot carry signing secrets securely and are not reproducible across macOS + Windows. One person's machine cannot be the signing oracle for both platforms. | Medium | GitHub Actions + tauri-action is the validated path for Tauri v2. Matrix build on `macos-latest` (for macOS .dmg) and `windows-latest` (for Windows .exe/.msi). tauri-action generates artifacts + latest.json and creates a GitHub Release. Trigger: push of a `v*` version tag (e.g., `v0.3.0`). All signing secrets injected via GitHub repo secrets. This is not optional complexity — it is the only mechanism that reliably signs both platforms in one workflow. |
| Private download page with invite-only access | Beta users need a URL. Email attachments fail on file size. Sharing a GitHub Release URL directly exposes the repository and bypasses any access control. | Low | A static HTML page or Notion page with a password or invite code, linking to the GitHub Release .dmg and .exe assets, is sufficient. No commercial beta distribution platform needed. Alternatives: Gumroad (free, supports file hosting + password protection), a simple Vercel-hosted static page. The page should display: minimum OS version, install instructions, and the feedback channel URL. |

## Differentiators

Features that improve the beta experience materially but are not blockers. Build these after table stakes are confirmed working.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Crash reporting via sentry-tauri | Know whether the app is crashing in production before users report it. Captures JS errors from the SolidJS frontend and Rust panics from the backend in a single Sentry issue stream. Without this, debugging user-reported issues requires asking users to reproduce and share logs — a high-friction process that most beta users skip. | Medium | sentry-tauri is a community plugin (author: timfish), not an official Sentry SDK. Sentry has confirmed no official Tauri SDK is on the roadmap. The plugin supports Tauri v2 and captures: JS errors, Rust panics, and native minidumps. It injects @sentry/browser into the WebView and bridges browser events to the Rust SDK for unified context. Sentry free tier: 5,000 errors/month. Verify sentry-tauri's Tauri v2 compatibility before committing — check the repository's open issues for v2-specific problems before investing time in integration. Alternative if sentry-tauri is fragile: structured local log file + a "Send logs" button in settings that uploads the log file to a private endpoint. |
| Privacy-first telemetry via Aptabase | Know if users are launching the app at all, completing queries, and using voice features. Without any telemetry, you cannot distinguish "the app crashes on launch" from "users installed it and stopped using it after day one" — two very different problems requiring opposite responses. | Low | Aptabase has a native tauri-plugin-aptabase crate. It is GDPR/CCPA/PECR compliant, collects no PII, uses no cookies or device fingerprinting, and is self-hostable. The Aptabase SDK makes a single anonymized event call per tracked event. Track: `app_launched`, `query_submitted`, `voice_input_used`, `step_completed`, `update_installed`. Disclose in beta onboarding and privacy notes. Aptabase has a free tier; self-hosting is free and straightforward on Railway or Fly.io. |
| In-app update notification dialog | By default, Tauri's updater checks silently. Users do not know an update installed unless told. A simple native dialog ("A new version is available — restart to update?") is already supported by tauri-plugin-updater with zero custom UI work. Without it, users may run stale versions for days after a fix ships. | Low | Set `"dialog": true` in the updater plugin configuration in tauri.conf.json. This activates a native OS dialog on update detection. The dialog is blocking and fires on app launch when an update is available. No frontend code required. |
| Structured feedback channel | Bug reports via email are unstructured, easy to miss, and create no record. A low-friction channel that beta users will actually use is worth the 30 minutes to set up. | Low | Recommended: Discord server with a #beta-feedback channel. Free, searchable, async, creates community around the product, and gives you a direct line to users for follow-up questions. Alternative: Typeform linked from the app tray menu or a link in the download page. A link from the app tray menu ("Send feedback") that opens the Discord invite or Typeform takes under an hour to implement and meaningfully increases feedback volume. |

## Anti-Features

Features to explicitly not build for v3.0. Each one is a scope trap.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Custom update server or update API | Writing and hosting a custom update endpoint adds infrastructure and maintenance burden with zero benefit over GitHub Releases for a closed beta. | Use tauri-action's `includeUpdaterJson: true`. The generated latest.json at a GitHub Releases URL is a complete, free, zero-maintenance update endpoint. |
| Crash reporting before signing pipeline is working | If the app cannot be installed cleanly, crash data is irrelevant. Instrumentation layered on a broken distribution is noise and wasted setup time. | Complete the signing + distribution pipeline first. Add Sentry in a second pass once users have a working install. |
| Public landing page or waitlist | A public waitlist is a marketing exercise, not a distribution requirement. It adds scope without closed beta value and implicitly commits you to a public launch timeline you haven't decided on yet. | A private download page with a password or invite token. Add a public landing page for the public launch milestone. |
| App Store submission (macOS or Windows) | macOS App Store review for a screen-capturing app requires specific entitlement justification, takes days to weeks, and blocks iteration speed. Windows Store review is a similar drag. Neither is necessary for a closed beta. | Distribute via Developer ID (macOS) and direct NSIS installer (Windows) outside any app store. |
| Custom installer or onboarding wizard | Beta users are typically technical enough to drag an app to /Applications or click through an NSIS wizard. A custom installer takes weeks and adds nothing for a beta of tens of users. | Ship the default Tauri bundle artifacts: .dmg for macOS (drag-to-Applications), NSIS .exe for Windows. |
| TestFlight (macOS) | TestFlight on macOS requires App Store Connect submission and review, applies to Mac Catalyst and native SwiftUI apps, and is not a natural fit for Tauri apps distributed outside the App Store. | Use Developer ID + direct download. |
| User accounts or auth within the app | The closed beta is access-controlled at the download link level (password or invite code). Building in-app accounts is a v4+ concern that would take weeks to implement correctly. | Gate access by the download link. User identity is not needed for closed beta. |
| Firebase App Distribution or equivalent mobile beta platforms | These platforms are designed for iOS/Android APK/IPA distribution. They do not natively support macOS .dmg or Windows .exe. Do not shoehorn a mobile tool into a desktop workflow. | GitHub Releases assets linked from a private download page. |
| Analytics dashboard or admin UI | Off-the-shelf (Aptabase, Sentry) gives you queryable data with zero build time. A custom dashboard is weeks of work for data you can query with a SQL interface in Aptabase or Sentry's existing UI. | Use Aptabase's built-in dashboard and Sentry's issue tracker. |

## Feature Dependencies

```
Cloudflare Worker live
  → Every AI feature is functional (all Claude, STT, TTS calls route through it)

Updater keypair generated before first build
  → `createUpdaterArtifacts: true` generates signed .sig files alongside installers
  → Signed build pipeline produces latest.json
  → Auto-updater endpoint live (GitHub Releases latest.json URL)
    → In-app update dialog fires on launch when new version detected

macOS Developer ID cert + notarization credentials in CI
  → Signed + notarized macOS .dmg
  → Gatekeeper does not block installation
  → Signed installer passes Tauri updater signature verification

Windows Trusted Signing or Azure Key Vault credentials in CI
  → Signed Windows .exe
  → SmartScreen warning reduced (not fully eliminated for new apps with low download volume)
  → Signed installer passes Tauri updater signature verification

GitHub Actions CI pipeline (macos-latest + windows-latest matrix)
  → Reproducible signed artifacts for both platforms
  → latest.json generated automatically by tauri-action
  → GitHub Release created with all assets at stable URLs

GitHub Release assets at stable URLs
  → Download page links to .dmg and .exe
  → Auto-updater endpoint (latest.json) is live

Private download page live
  → Beta users can get an installer URL

[Optional — add after above is stable and verified]
Aptabase app key in CI / build env
  → Basic telemetry active in production builds

Sentry DSN in CI / build env (if sentry-tauri is stable on Tauri v2)
  → Crash reporting active in production builds

Discord server created
  → Feedback channel link included in download page and tray menu
```

## MVP Recommendation: Phased Execution for v3.0

Ship in this order. Do not proceed to the next phase until the previous one is verified end-to-end.

**Phase A — Backend live (1–2 days)**
1. Create KV namespace in Cloudflare dashboard; paste namespace ID into wrangler.toml
2. Set API key secrets via `wrangler secret put` (Claude API key, AssemblyAI key, ElevenLabs key, HMAC secret)
3. Run `wrangler deploy` to production
4. Verify: invoke overlay, submit a query, confirm AI response arrives through deployed worker

**Phase B — Signing credentials provisioned (1–2 days)**
5. Generate updater keypair: `cargo tauri signer generate` → save both keys; put pubkey in tauri.conf.json, put private key in GitHub secret `TAURI_SIGNING_PRIVATE_KEY`
6. Enroll in Apple Developer Program if not already; create "Developer ID Application" certificate; export as .p12; base64-encode; set `APPLE_CERTIFICATE` and `APPLE_CERTIFICATE_PASSWORD` GitHub secrets
7. Create App Store Connect API key for notarization (preferred over Apple ID + app-specific password — more stable in CI); set `APPLE_API_ISSUER` and `APPLE_API_KEY` GitHub secrets
8. Create Microsoft Azure account; enroll in Microsoft Trusted Signing ($9.99/month); configure `trusted-signing-cli` credentials as GitHub secrets

**Phase C — CI pipeline + first signed build (2–3 days)**
9. Write GitHub Actions workflow: matrix on macos-latest + windows-latest, runs tauri-action, passes signing env vars, enables `includeUpdaterJson: true`, creates a draft GitHub Release
10. Push a test version tag (e.g., `v0.1.0-beta.1`); confirm pipeline succeeds on both platforms
11. Verify: .dmg opens on macOS without Gatekeeper warning; .exe installs on Windows with reduced SmartScreen friction; latest.json is present in the release

**Phase D — Distribution endpoint + install test (1 day)**
12. Verify latest.json URL is stable and returns correct JSON format
13. Set `endpoints` in tauri.conf.json to the GitHub Releases latest.json URL with `{{target}}` / `{{arch}}` interpolation
14. Create private download page with password protection; link to .dmg and .exe release assets
15. Full end-to-end test: fresh machine install from download page → app opens → AI responds through deployed worker → push v0.1.1-beta.1 → app auto-updates on next launch

**Phase E — Observability (1–2 days, after Phase D is verified)**
16. Add Aptabase: instrument `app_launched`, `query_submitted`, `voice_input_used` events; verify events appear in Aptabase dashboard from a test install
17. Evaluate sentry-tauri v2 compat; add crash reporting if confirmed stable
18. Create Discord server; add "Send feedback" link in tray menu or settings; include in download page

## Complexity Summary

| Capability | Effort | Blocker if Missing |
|------------|--------|-------------------|
| Cloudflare Worker deploy | Low (~1 day) | Yes — app non-functional |
| macOS signing + notarization | Medium (1–2 days) | Yes — Gatekeeper hard-blocks install |
| Windows code signing | Medium (1–2 days) | Partial — SmartScreen warns but doesn't hard-block |
| Updater keypair + endpoint | Medium (1 day) | Soft — manual reinstall is painful but possible |
| GitHub Actions CI pipeline | Medium (2–3 days) | Yes — signing requires CI |
| Private download page | Low (hours) | Yes — users need a URL |
| In-app update dialog | Low (30 min config) | No — improves UX, not a blocker |
| Aptabase telemetry | Low (hours) | No — observability, not function |
| Crash reporting (sentry-tauri) | Medium (1 day, if stable) | No — observability, not function |
| Feedback channel (Discord) | Low (30 min) | No — improves feedback quality |
